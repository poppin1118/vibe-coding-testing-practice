---
description: 審查程式碼品質（變數宣告、SOLID 原則、設計模式）並產出報告
---

扮演一位資深軟體工程師，針對使用者指定的檔案或資料夾進行 Code Review。
若使用者未指定範圍，則以 `src/` 為預設審查範圍。

---

## STEP 1：確認審查範圍

1. 確認使用者指定的審查目標（檔案 / 資料夾 / 整個 `src/`）
2. 列出所有待審查的檔案清單，讓使用者確認後才繼續

---

## STEP 2：靜態分析（自動化工具）

1. 執行 `npm run lint` 收集 ESLint 結果
2. 執行 `npx tsc --noEmit` 收集 TypeScript 編譯錯誤
3. 將工具產出的問題整理為結構化清單，後續併入報告

---

## STEP 3：逐檔人工審查

對每一個待審查檔案，依下列 **五個維度** 進行審查。
每個維度只需記錄「有問題」的項目，沒問題的不用列出。

### 3-1. 變數與函數宣告

| 項目 | 說明 |
|------|------|
| `const` 優先 | 能用 `const` 的地方不該用 `let`；禁止使用 `var` |
| 命名語意 | 變數、函數名稱是否清楚表達意圖（避免 `data`, `temp`, `flag` 等模糊名稱） |
| 函數長度 | 單一函數是否過長（建議 ≤ 30 行），是否可拆分 |
| 參數數量 | 函數參數是否過多（建議 ≤ 3 個），過多時應使用物件參數 |
| Magic Number | 是否存在未命名的魔術數字或字串，應抽為具名常數 |

### 3-2. SOLID 原則

| 原則 | 檢查重點 |
|------|----------|
| **S** – 單一職責 | 每個元件 / 函數 / 模組是否只負責一件事 |
| **O** – 開放封閉 | 擴展新功能時是否需要修改現有程式碼（應透過抽象擴展） |
| **L** – 里氏替換 | 子型別是否能安全替換父型別使用（interface / 繼承是否合理） |
| **I** – 介面隔離 | Props / Interface 是否被迫依賴用不到的屬性 |
| **D** – 依賴反轉 | 高階模組是否直接依賴低階實作（應依賴抽象 / 介面） |

### 3-3. 設計模式與架構

| 項目 | 說明 |
|------|------|
| 適用 Pattern | 現有實作是否有更合適的設計模式可套用（例如 Strategy、Observer、Factory、Compound Component 等） |
| 重複邏輯 | 是否有相似邏輯散落多處，可抽取為 Custom Hook 或共用工具函數 |
| 狀態管理 | 是否有 prop drilling 問題、狀態是否放在合理的層級 |
| 關注點分離 | UI 邏輯與業務邏輯是否適當分離（例如 Hook 抽離邏輯、元件只負責渲染） |

### 3-4. 可讀性與可維護性

| 項目 | 說明 |
|------|------|
| 註解品質 | 是否有需要但缺少的註解；是否有過時或誤導的註解 |
| 程式碼結構 | import 排序、檔案組織是否一致 |
| 型別安全 | 是否濫用 `any`；型別定義是否完整 |
| 錯誤處理 | 是否有未處理的例外或被忽略的 Promise rejection |

### 3-5. 效能與安全

| 項目 | 說明 |
|------|------|
| 不必要的重新渲染 | 是否缺少 `React.memo`、`useMemo`、`useCallback` 的合理使用 |
| 記憶體洩漏 | useEffect 是否有正確清理（cleanup function） |
| XSS 風險 | 是否有使用 `dangerouslySetInnerHTML` 或其他不安全的 DOM 操作 |

---

## STEP 4：產出審查報告

1. 在 `doc/code-review/` 目錄下建立報告（若目錄不存在則建立）
2. 檔名格式：`review-YYYY-MM-DD-<簡短描述>.md`
3. 報告格式如下：

```markdown
# Code Review 報告

- **審查日期**：YYYY-MM-DD
- **審查範圍**：（列出檔案清單或資料夾）
- **審查人**：AI Assistant

---

## 摘要

| 嚴重度 | 數量 |
|--------|------|
| 🔴 嚴重（Must Fix） | N |
| 🟡 建議（Should Fix） | N |
| 🟢 可選（Nice to Have） | N |

---

## 靜態分析結果

（STEP 2 的 ESLint / TypeScript 問題）

---

## 審查明細

### 📄 `path/to/file.tsx`

#### 🔴 嚴重

- **[變數宣告]** 第 XX 行：說明問題 → 建議做法
- **[SOLID-S]** 說明問題 → 建議做法

#### 🟡 建議

- **[設計模式]** 說明問題 → 建議做法

#### 🟢 可選

- **[可讀性]** 說明問題 → 建議做法

（每個檔案重複此結構）

---

## 總結與建議

（整體架構層面的綜合建議，優先處理順序）
```

---

## STEP 5：確認與討論

1. 將報告呈現給使用者
2. 詢問是否需要針對特定問題進一步說明或直接修復
3. 若使用者同意修復，依嚴重度由高到低逐一處理，每次修復後執行相關測試確認無破壞

---

**重要規則：**
- 審查過程中不要自行修改任何程式碼，僅記錄問題
- 所有問題必須標註嚴重度（🔴 / 🟡 / 🟢）
- 建議做法必須是具體可執行的，附上程式碼片段，不要只說「建議優化」
- 若某個維度完全沒問題，在報告中簡短標註「✅ 無異常」即可
